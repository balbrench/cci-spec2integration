---
description: [Manual] Compile the integration IR to a Logic Apps Standard solution (workflows, connections, Bicep, tests, CI/CD).
argument-hint: [spec-folder]
allowed-tools: Read, Edit, Write, Grep, Glob, Agent
---

Run the Azure pack's implementation chain.

Steps:
1. Resolve the integration folder (argument or most recently modified `specs/*/*/`).
2. Verify `.spec2integration/state.json` has `activePlatform: azure`; otherwise stop.
3. Verify `tasks.md` exists; otherwise instruct the user to run `/plan` and `/tasks`.
4. Read `integration-ir.yaml` and resolve the run mode before invoking any compiler.
   - **Brownfield / migration mode**: `metadata.scenario == migration` and a top-level `source:` block is present.
   - **Greenfield mode**: `metadata.scenario == greenfield` (or omitted) and there is no top-level `source:` block.
   - If the IR mixes those signals, stop with a clear error. The Azure pack supports both modes, but only **one mode at a time** for a given `/implement-azure` invocation.
5. Run the `ir-validator` agent on the integration folder. If it returns any errors, stop and report them - do not proceed to compilation with an invalid IR.
6. Resolve the output directories. The integration folder mixes pipeline artifacts (spec, IR, contracts, reports) with the deployable Azure solution. To keep them separate, the Azure pack writes its output to **subfolders** of the integration folder, per `.claude/skills/logicapp-standard-layout/SKILL.md`.
   - **Logic Apps Standard project root** = `<integration-folder>/app/` - contains `host.json`, per-flow folders (NO `src/` wrapper), `connections.json`, `parameters.*.json`, `appsettings.*.json`, `local.settings.json`, `workflow-designtime/`, `Artifacts/Maps/`, `Artifacts/DataMapper/`, `Artifacts/Liquid/`, `Artifacts/Schemas/`, `lib/custom/net8/` (drop folder for sibling `Functions/` DLLs), `.vscode/`, `.funcignore`, `.gitignore`, `identity-role-assignments.json`, `global.json`, and the `.github/workflows/` folder.
   - **Sibling Functions project** = `<integration-folder>/Functions/` - a separate .NET 8 in-process WebJobs SDK project at the integration root (NOT inside the logic-app folder). Its csproj's `LogicAppFolderToPublish` resolves to `..\app` so DLLs land in `app/lib/custom/net8/` after `dotnet build`. This project holds in-process custom code (`InvokeFunction` targets) for Logic Apps Standard flows only - it is not the same as a stand-alone Function App.
   - **Stand-alone Function App projects** = `<integration-folder>/FunctionApps/<FlowName>/` - one .NET 8 isolated-worker project per IR flow whose `implementation.host == function-app`. Sibling to `app/` and to `Functions/`; deployed as separate Azure Function Apps. Emitted by `azure-functions-compiler`. Skipped when no flow routes to `function-app`.
   - **Data Factory artifacts** = `<integration-folder>/adf/` - pipelines, datasets, linked services, triggers, and optionally data flows for every IR flow whose `implementation.host == data-factory`. Sibling to `app/`, `Functions/`, and `FunctionApps/`; deployed via the factory module under `infra/`. Emitted by `azure-data-factory-compiler`. Skipped when no flow routes to `data-factory`.
   - **Bicep** = `<integration-folder>/infra/` - outside the logic-app folder so it is not bundled into the func deploy package; `azd` reads it from one level above.
   - **MSTest project** = `<integration-folder>/tests-mstest/` - separated from `app/tests/` (which is reserved for the designer's JSON request fixtures).
   - **azd manifest** = `<integration-folder>/azure.yaml` - at the integration folder root, with `infra.path: ./infra`. For Logic Apps Standard, do **not** add `services.logicapp.project: ./app`; `azd` provisions infra only, and the Logic App content under `app/` is package-deployed separately.
   - **Workspace file** = `<integration-folder>/<slug>.code-workspace` - opens `./app` in VS Code so the Azure Logic Apps Standard extension attaches to the deployable project; also include `./Functions` as a second folder entry if the integration has a sibling custom-code project, so developers can edit the C# source alongside the workflows.
   When invoking each agent below, pass the appropriate output directory explicitly. Do not let agents emit at the integration folder root except for `azure.yaml`, the `.code-workspace`, and `azure-review.md`.
7. Bucket every IR flow by `implementation.host` (default `logic-app-standard` when the field is absent). Then, in order, invoke:

   **Ordering is a hard dependency, not a suggestion — do NOT parallelize across a read-after-write seam.** The independent per-bucket compilers (`azure-logic-apps-compiler`, `azure-functions-compiler`, `azure-data-factory-compiler`) may run concurrently with each other, but the downstream agents consume their outputs and MUST run strictly after:
   - `azure-bicep-author` **MUST run after `azure-connections-binder` has finished writing `app/connections.json`** — it mirrors the exact `@appsetting('NAME')` names from `connections.json` into the Logic App site settings (step 6/4a of its agent). If bicep runs first (or in parallel) it reads a pre-binder/bare `connections.json`, invents its own app-setting names, and the deployed MI connections resolve to null endpoints (a deploy-blocking Critical the reviewer otherwise has to catch by manual cross-check).
   - `azure-workflow-tester` and `azure-cicd-author` MUST run after the artifacts they reference exist (`app/`, `tests-mstest/`, `adf/`).
   Treat the binder→bicep edge as serial every run.

   - `azure-logic-apps-compiler` (output `<integration-folder>/app/`) - runs for the `logic-app-standard` bucket. Emits `<FlowName>/workflow.json` per flow at the project root, `host.json`, `workflow-designtime/`, `Artifacts/Maps/<MapName>.xsl`, `Artifacts/DataMapper/<MapName>.lml`, `Artifacts/Liquid/<MapName>.liquid`, `Artifacts/Schemas/<SchemaName>.xsd`, `.funcignore`, `.gitignore`. **Skip this agent only if the `logic-app-standard` bucket is empty.**
   - `azure-functions-compiler` (output `<integration-folder>/FunctionApps/<FlowName>/` per flow) - runs for the `function-app` bucket. Emits one stand-alone .NET 8 isolated-worker project per flow (`<FlowName>.csproj`, `Program.cs`, `host.json`, `local.settings.json`, `Functions/<FlowName>Function.cs`, `Functions/Activities/`, `Functions/Orchestrators/` when `durablePattern` is set, `Models/`, `Mappings/`, `Helpers/`). **Skip this agent only if the `function-app` bucket is empty.**
   - `azure-data-factory-compiler` (output `<integration-folder>/adf/`) - runs for the `data-factory` bucket. Emits `pipelines/<FlowName>.json`, `datasets/`, `linkedServices/`, `triggers/`, `dataflows/` (when transforms warrant Mapping Data Flow), `integrationRuntimes/` (when Self-Hosted IR is required), `factory/<factoryName>.json`, and `README.md`. **Skip this agent only if the `data-factory` bucket is empty.**
   - `azure-local-functions-author` (output `<integration-folder>/Functions/` - sibling project, NOT inside the logic-app folder, NOT the same as `FunctionApps/`) - scans the **Logic Apps Standard** workflows emitted above for `InvokeFunction` targets and emits `<ProjectName>.csproj` (TFM `net8`, in-proc WebJobs SDK), `<FunctionName>.cs` per target (instance class with `[Function]` from `Microsoft.Azure.Functions.Worker` + `[WorkflowActionTrigger]`), `Models/*.cs`, and `Helpers/*.cs`. The csproj's post-build target publishes DLLs to `<integration-folder>/app/lib/custom/net8/`. **Skip this agent only if step 1's scan finds zero `InvokeFunction` actions across all workflows.**
   - `azure-connections-binder` (output `<integration-folder>/app/`) - emits `connections.json`, `parameters.json`, `appsettings.dev.json`, `appsettings.prod.json`, `local.settings.json`, `identity-role-assignments.json`, `.vscode/`, `global.json`. Only runs when the `logic-app-standard` bucket is non-empty.
   - `azure-bicep-author` (output `<integration-folder>/infra/` for modules, `<integration-folder>/azure.yaml` for the azd manifest) - emits `infra/` modules and `azure.yaml`. The manifest's `services` block reflects only the buckets that `azd` deploys directly: one `services.<flowName>.project: ./FunctionApps/<FlowName>` entry per stand-alone Function App. Logic Apps Standard content under `app/` is provisioned via `infra/` and then package-deployed separately; Data Factory artifacts are wired in by the `dataFactory.bicep` module rather than as an azd service.
   - `azure-workflow-tester` (output `<integration-folder>/tests-mstest/`) - emits the MSTest project with `WorkspacePath` pointing at the relative path to `app/` from each test project. Only runs when the `logic-app-standard` bucket is non-empty.
   - `azure-cicd-author` (output `<integration-folder>/app/.github/workflows/`) - emits the GitHub Actions workflows. The pipeline includes one publish job per non-empty bucket (Logic App, each Function App, and the Data Factory artifact import).
   - `azure-reviewer` (output `<integration-folder>/azure-review.md`) - audits every emitted artifact tree.
   After the compilers run, also write `<integration-folder>/<slug>.code-workspace`. The `folders` array MUST contain `./app` (when present), `./Functions` (when present), and one entry per `./FunctionApps/<FlowName>` that was emitted. Without this, the C# / Function App projects are invisible in the VS Code Explorer.
8. After each agent, surface any TODOs or Bicep comment blocks it emitted.
8a. **Bicep compile gate (hard — run this; the `azure-bicep-author`/`azure-reviewer` agents have no Bash and cannot).** After `azure-bicep-author` returns, run `az bicep build --file <integration-folder>/infra/main.bicep` (and on each module if `main.bicep` only references them). If it exits non-zero with any `Error BCP...`, the template does not compile and **every** deployment path is broken: re-invoke `azure-bicep-author` with the exact error (file:line + BCP code) to fix it, then re-compile. Do not proceed to the reviewer/secret-scan/verdict until `az bicep build` exits 0. Delete the transient `infra/main.json` the build emits. (This gate exists because the agents can only emit/grep, not compile — a space-separated `@allowed([ 'A' 'B' ])` / `BCP236` shipped undetected before it was added.) Also confirm `azure-bicep-author` emitted `infra/main.parameters.json` (azd reads it, not `.bicepparam`); if missing, bounce it back — `azd provision` will otherwise hang on the no-default params.
8b. **ADF artifact JSON-validity gate (hard — run this; the compiler/reviewer agents do not strict-parse their own JSON).** When the `data-factory` bucket is non-empty, after `azure-data-factory-compiler` returns, strict-parse **every** file under `<integration-folder>/adf/**/*.json` (e.g. PowerShell `Get-ChildItem adf -Recurse -Filter *.json | % { Get-Content $_ -Raw | ConvertFrom-Json | Out-Null }`, or `python -c "import json,glob; [json.load(open(f)) for f in glob.glob('adf/**/*.json',recursive=True)]"`). If any file fails to parse, the ADF artifact is undeployable: re-invoke `azure-data-factory-compiler` with the exact file + line/col + parser message to fix it, then re-parse. Do not proceed to the reviewer/verdict until every `adf/*.json` parses. (This gate exists because a systematic emitter bug — an extra `}` when a Web-activity `body`+`typeProperties` are closed inline as `} }` — shipped invalid pipeline JSON undetected until `az datafactory` import; nothing else in the pipeline strict-parses ADF artifacts.)
9. If the `azure-reviewer` verdict is `BLOCKED`, stop and print all Critical findings. Do not proceed further - the solution must not be considered complete while Critical findings exist.
10. **Only when the source IR is reverse-engineered** (`<integration-folder>/integration-ir.yaml` has a top-level `source:` block, typically `source.platform: biztalk`), call the `secret-scanner` agent with the full list of directories written by agents in step 7: `<integration-folder>/app/` (when present), `<integration-folder>/Functions/` (when present), `<integration-folder>/FunctionApps/` (when present - scans every `FunctionApps/<FlowName>/` project), `<integration-folder>/adf/` (when present - scans pipelines, datasets, linked services, triggers), `<integration-folder>/infra/`, `<integration-folder>/tests-mstest/` (when present), plus the top-level files `<integration-folder>/azure.yaml` and `<integration-folder>/<slug>.code-workspace`. If it returns `BLOCKED`, print its Sev-1 findings. Skip this step for greenfield: the `azure-connections-binder` and `azure-bicep-author` agents emit only managed-identity and Key Vault references by design, and the `azure-reviewer` audit in step 7 already enforces that no inline secrets appear in the generated artifacts.
11. Compute SHA-256 hashes of every file written by the agents in step 7 (across every emitted bucket - `app/`, `Functions/`, `FunctionApps/<FlowName>/`, `adf/`, `infra/`, `tests-mstest/`, plus the top-level `azure.yaml` and `.code-workspace`). On Windows use PowerShell `Get-FileHash -Algorithm SHA256`; on Linux/macOS use `sha256sum`. Store the baseline in the **per-integration** `<integration-folder>/status.json` (not in the global `.spec2integration/state.json`): read the existing `status.json`, add or replace the top-level keys `artifactHashes` (flat object mapping repo-relative file path to hex digest) and `lastImplement` (`{ "completedAt": "<ISO-8601>", "flowCount": N, "host": "<logic-app-standard|function-app|data-factory>", "verdict": "<PASS|BLOCKED>", "artifactCount": N }`), then write it back preserving all other keys. This baseline is read by `/drift-check`.
12. Final step: print the Azure Reviewer verdict and counts, the secret-scan verdict (or `skipped (greenfield)` when step 10 was skipped), the per-bucket flow counts (`<L> logic-app-standard / <F> function-app / <D> data-factory`), and a local development quick-start:
    ```
    To run locally:
    1. Open <integration-folder>/<slug>.code-workspace in VS Code (or just open <integration-folder>/app/ as a folder).
    2. Start Azurite: Ctrl+Shift+P -> "Azurite: Start"
    3. Press F5 (or run: `cd app && func host start`)
    4. Logic App available at http://localhost:7071
    5. For stand-alone Function Apps: `cd FunctionApps/<FlowName> && func start`
    6. For Data Factory artifacts: deploy via `azd up` - there is no local ADF runtime.
    ```
13. Refresh `<integration-folder>/status.json` per `.claude/skills/pipeline-status/SKILL.md`. Mark stage 10 (Implement) as `done` (or `blocked` if the azure-reviewer verdict was `BLOCKED`, or if the secret scan ran and returned Sev-1) with summary `<L>/<F>/<D> flows generated (logic-app/function-app/data-factory) under <integration-folder>/`. Mark stage 11 (Tests) as `done` once `<integration-folder>/tests-mstest/**/*.csproj` exists.
14. Read the refreshed `<integration-folder>/status.json` and ask one short follow-up question in chat: `What do you want to do next?`
   - If the chat surface supports selectable options, offer them as choices.
   - Otherwise render a numbered list and wait for the user's reply.
   - Print `Why: <next.reason>` before the choices.
   - Offer these choices:
     - `Run recommended step: <next.command>`
     - `Fix azure-review.md or secret-scan findings, then re-run /implement-azure <integration-folder>`
     - `Stop here`
